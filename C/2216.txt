       <h3 id="replacing-a-functional-node"><a href="#replacing-a-functional-node">Replacing a functional node</a></h3> 
       <ol> 
        <li> <p><a href="/en/enterprise-server@3.1/admin/guides/installation/setting-up-a-github-enterprise-server-instance">Provision and install GitHub Enterprise Server</a> with a unique hostname on the replacement node.</p> </li> 
        <li> <p>Using the administrative shell or DHCP, <strong>only</strong> configure the IP address of the replacement node. Don't configure any other settings.</p> </li> 
        <li> <p>To add the newly provisioned replacement node, on any node, modify the <code>cluster.conf</code> file to remove the failed node and add the replacement node. For example, this modified <code>cluster.conf</code> file replaces <code>ghe-data-node-3</code> with the newly provisioned node, <code>ghe-replacement-data-node-3</code>:</p> <pre>[cluster "<em>ghe-replacement-data-node-3</em>"]
  hostname = <em>ghe-replacement-data-node-3</em>
  ipv4 = <em>192.168.0.7</em>
  # ipv6 = fd12:3456:789a:1::7
  git-server = true
  pages-server = true
  mysql-server = true
  elasticsearch-server = true
  redis-server = true
  memcache-server = true
  metrics-server = true
  storage-server = true
</pre> </li> 
        <li> <p>From the administrative shell of the node with the modified <code>cluster.conf</code>, run <code>ghe-cluster-config-init</code>. This will initialize the newly added node in the cluster.</p> </li> 
        <li> <p>From the same node, run <code>ghe-cluster-config-apply</code>. This will validate the configuration file, copy it to each node in the cluster, and configure each node according to the modified <code>cluster.conf</code> file.</p> </li> 
        <li> <p>If you're taking a node offline that has <code>git-server = true</code> set in cluster.conf, evacuate the node. For more information, see "<a href="/en/enterprise-server@3.1/admin/clustering/evacuating-a-cluster-node">Evacuating a cluster node</a>."</p> </li> 
        <li> <p>To mark the failed node offline, on any node, modify the <a href="/en/enterprise-server@3.1/admin/guides/clustering/initializing-the-cluster/#about-the-cluster-configuration-file">cluster configuration file</a> (<code>cluster.conf</code>) in the relevant node section to include the text <code>offline = true</code>.</p> <p>For example, this modified <code>cluster.conf</code> will mark the <code>ghe-data-node-3</code> node as offline:</p> <pre>  [cluster "ghe-data-node-3"]
  hostname = ghe-data-node-3
  <em>offline = true</em>
  ipv4 = 192.168.0.6
  # ipv6 = fd12:3456:789a:1::6
  </pre> </li> 
        <li> <p>From the administrative shell of the node where you modified <code>cluster.conf</code>, run <code>ghe-cluster-config-apply</code>. This will validate the configuration file, copy it to each node in the cluster, and mark the node offline.</p> </li> 
        <li> <p>If you're replacing the primary MySQL or Redis node, in <code>cluster.conf</code>, modify the <code>mysql-master</code> or <code>redis-master</code> value with the replacement node name.</p> <p>For example, this modified <code>cluster.conf</code> file specifies a newly provisioned cluster node, <code>ghe-replacement-data-node-1</code> as the primary MySQL and Redis node:</p> <pre>mysql-master = <em>ghe-replacement-data-node-1</em>
redis-master = <em>ghe-replacement-data-node-1</em>
</pre> </li> 
       </ol>
