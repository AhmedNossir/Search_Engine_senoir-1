       <h1 class="devsite-page-title">Complying with YouTube's Developer Policies<devsite-bookmark></devsite-bookmark></h1> <devsite-toc class="devsite-nav" devsite-toc-embedded> 
       </devsite-toc> 
       <div class="devsite-article-body clearfix
    "> 
        <p>If you use YouTube's API services, you must adhere to our:</p> 
        <ul> 
         <li><a href="/youtube/terms/developer-policies">Developer Policies</a></li> 
         <li><a href="https://support.google.com/youtube/answer/9142671">Content Manager Policies</a></li> 
         <li><a href="https://developers.google.com/youtube/terms/api-services-terms-of-service">YouTube API Services Terms of Service</a></li> 
         <li><a href="https://www.youtube.com/about/policies/#community-guidelines">Community Guidelines</a></li> 
        </ul> 
        <p>As a developer, you must familiarize yourself with these policies. This article provides examples that provide additional clarifications about specific policies, and answers some frequently asked questions.</p> 
        <aside class="note">
         <strong>Note:</strong><span> Violating any of these policies may result in reduction of your API quota, revoking your API keys and their associated privileges, or other actions, including termination of the Google account associated with the API service found to be in violation.</span>
        </aside> 
        <p>If, after reviewing this article and the policies linked to above, you’re unsure whether your service is allowed, please apply for an <a href="https://support.google.com/youtube/contact/yt_api_form">API Compliance Audit</a>, and include a clear summary that mentions any end users in the audit form.</p> 
        <h2 id="respect_user_privacy" data-text="Respect user privacy.">Respect user privacy.</h2> 
        <p><strong>What this means:</strong> Don’t violate user privacy, harvest user data, or use the API to surveil users. Your app must have a privacy policy that protects users and complies with Google’s <a href="https://policies.google.com/privacy">privacy policies</a>. Once you have a user’s permission to access or store their data, that user must continue to have control over what happens to that data. You need to make it easy for users to request deletion of their personal data. More information can be found <a href="/youtube/terms/developer-policies#a.-api-client-terms-of-use-and-privacy-policies">here</a>.</p> 
        <h3 id="examples" data-text="Examples">Examples</h3> 
        <p><strong>Don’t use YouTube’s API to:</strong></p> 
        <ul> 
         <li>Harvest, track, infer, derive or store information that can be used to identify a user without their consent. Examples include (this is not a complete list): 
          <ul> 
           <li>Full names or usernames</li> 
           <li>Passwords</li> 
           <li>Facial recognition data</li> 
           <li>Contact information including email or phone number</li> 
           <li>Online activity such as browsing history 
            <ul> 
             <li>Example: Using YouTube’s API to create an app that tracks a user’s viewing history, location or browsing habits without their knowledge or consent.</li> 
            </ul></li> 
           <li>Harvest, track, infer, derive or store the following about a user without their consent. Examples include (this is not a complete list): 
            <ul> 
             <li>Health information</li> 
             <li>Gender identity</li> 
             <li>Sexual orientation</li> 
             <li>Political affiliation</li> 
             <li>Religious affiliation or belief</li> 
             <li>Race or ethnicity</li> 
             <li>Immigration status</li> 
             <li>Financial status</li> 
             <li>Criminal history</li> 
             <li>Trade union membership or organization</li> 
            </ul></li> 
           <li>Facilitate surveillance -- for example, tracking their location, browsing history or other online activity without their consent.</li> 
           <li>Restrict, filter, or prohibit a user’s access to content on YouTube without their knowledge or consent.</li> 
           <li>Allow any unauthorized third parties to access, use, or download any data noted here.</li> 
           <li>Store a user’s information indefinitely. If a user requests that you delete their data or if you’re unable to verify their authorization, you must do so within 30 days.</li> 
           <li>Use, request, or store a user’s login credentials (username and password)</li> 
          </ul></li> 
        </ul> 
        <h2 id="only_offer_metrics_that_are_available_via_youtube’s_api_services" data-text="Only offer metrics that are available via YouTube’s API services.">Only offer metrics that are available via YouTube’s API services.</h2> 
        <p><strong>What this means:</strong> Don’t use YouTube’s API to offer independently calculated or derived metrics or data that replace or provide new data that isn’t available via YouTube’s API services. More information can be found <a href="/youtube/terms/developer-policies#e.-handling-youtube-data-and-content">here</a>.</p> 
        <h3 id="examples_2" data-text="Examples">Examples</h3> 
        <p><strong>Don’t use YouTube’s API to:</strong></p> 
        <ul> 
         <li>Display metrics that replace those offered by YouTube’s API services.</li> 
         <li>Display metrics that inaccurately reflect those offered by YouTube’s API services.</li> 
         <li>Combine data from YouTube’s API with data from other sources. If you provide data that is derived from sources other than YouTube’s API alongside data that you get from YouTube’s API, you must make the difference clear to the user. 
          <ul> 
           <li><strong>Example:</strong> Providing a "user engagement" metric that includes engagement from YouTube in combination with other platforms.</li> 
          </ul></li> 
         <li>Display data from YouTube next to data from other platforms without clarifying the difference between the types and sources of data.</li> 
         <li>Compile or aggregate authorized API data unless you are making the compiled API data viewable ONLY to the content or channel owner or one of their authorized representatives.</li> 
         <li>Gain insights into the number of users, number of videos uploaded, watch time, financial performance, or any other aspect of YouTube’s business.</li> 
         <li>Make any claims on whether a video or channel is safe or suitable to watch or advertise against.</li> 
         <li>Estimate the watch time or unique reach of a channel or video.</li> 
         <li>Estimate the number of paid views, sponsored views, or average advertising CPM of a video.</li> 
         <li>Estimate audience affinities, demographics, or audience composition of a channel or video.</li> 
         <li>Infer or estimate the content category/type of a video or channel; you may only use the content type returned by the YouTube API.</li> 
         <li>Estimate the monetization status of a video or channel, or make claims on whether a video or channel should be monetized.</li> 
         <li>Merge or combine YouTube API data with any other data.</li> 
         <li>Return information like the total number of video views and offer a number different from the number provided by YouTube’s API.</li> 
         <li>Infer or project financial performance of a YouTube channel.</li> 
         <li>Gamify channel performance by ranking or tracking views between different channels, or generally stoking creator rivalries.</li> 
         <li>Estimate viewer satisfaction or dissatisfaction with a particular YouTube channel.</li> 
         <li>Calculate and assign custom "scores" to channels based on independently calculated averages or ratios -- for example, average view count, comment count, or overall brand suitability.</li> 
        </ul> 
        <h3 id="acceptable_metrics" data-text="Acceptable metrics">Acceptable metrics</h3> 
        <p>Acceptable metrics are those that use only YouTube API data and combine them via simple mathematical calculations (combine them via addition, subtraction, averages, multiplication, division). These metrics must not incorporate any other external data sources. This allows us to ensure that the represented data is accurate.</p> 
        <p><strong>Examples</strong></p> 
        <ul> 
         <li>Average daily views in a month</li> 
         <li>Average video duration</li> 
         <li>Number of subscribers gained or lost</li> 
         <li>Average number of new subscribers in a month</li> 
         <li>Total views in a group of videos/channels</li> 
         <li>Top viewed videos/channels sorted by views, likes/dislikes, subscribers</li> 
         <li>Graphs visualizing raw metrics, e.g: increase in views, subscribers, likes/dislikes</li> 
        </ul> 
        <h2 id="your_api_service_must_reflect_a_user’s_standard_experience_on_youtube" data-text="Your API service must reflect a user’s standard experience on YouTube.">Your API service must reflect a user’s standard experience on YouTube.</h2> 
        <p><strong>What this means:</strong> Any service using the YouTube API can’t diminish or remove features that are part of a user’s standard experience on YouTube, such as captions, volume controls, etc. More information can be found <a href="/youtube/terms/developer-policies#f.-user-experience">here</a>.</p> 
        <h3 id="examples_3" data-text="Examples">Examples</h3> 
        <p><strong>Don’t use YouTube’s API to:</strong></p> 
        <ul> 
         <li>Modify, add to or block the standard playback function of the YouTube video player. Some examples include: 
          <ul> 
           <li>Blocking a link that would normally appear in the YouTube player from appearing in your application.</li> 
           <li>Disabling or blocking Related Video links from appearing after the video completes.</li> 
           <li>Removing or altering the video metadata. In general, video metadata such as thumbnail and title must be visible to the viewer and unmodified. The video thumbnail must not be altered. 
            <ul> 
             <li><strong>Note:</strong> Custom play buttons over the YouTube thumbnail are acceptable, but tapping must initiate playback.</li> 
            </ul></li> 
           <li>Links must open in the YouTube application whenever the application is available on a user’s device, or if not installed, via the system web browser.</li> 
           <li>Blocking the standard features of a YouTube player (like the settings wheel) from appearing in your API service.</li> 
           <li>Overriding the platform-specific rendering of the YouTube video player. 
            <ul> 
             <li><strong>Example:</strong> Mobile optimized UI must appear on mobile apps and devices.</li> 
            </ul></li> 
           <li>Restricting ads from playing in your API service when they would otherwise play on YouTube or in an embedded video.</li> 
           <li><strong>Note:</strong> Overlays for the purposes of obtaining user consent or playback controls (e.g. mute, full screen, play, pause, etc.) are acceptable so long as they do not conflict with the YouTube player UI elements.</li> 
          </ul></li> 
         <li>Restrict YouTube’s ability to verify from where the playback occurs. 
          <ul> 
           <li><strong>Example:</strong> In the case of mobile applications using a WebView to host the YouTube IFrame SDK web player, failing to accurately represent the name of your application (e.g. com.company.appname) accurately as HTTP Referer header.</li> 
           <li><strong>Example:</strong> Interfering with other playback context information necessary for view verification (including cookies) being sent to YouTube. 
            <ul> 
             <li>For privacy sensitive developers who deem it necessary, a user consent flow with link to Google’s Privacy Policy is acceptable.</li> 
            </ul></li> 
          </ul></li> 
         <li>Apply any restrictions or block access to a user watching a video. If a user has to do anything besides clicking the play button, there’s a good chance that you’re violating this policy. An example includes: 
          <ul> 
           <li><strong>Example:</strong> Restricting access to a video by requiring a user to complete a survey, download an app, subscribe to a channel, share a video on a social media platform, leave comments, or do anything other than click the "Play" button in order to watch a video they chose to watch.</li> 
          </ul></li> 
         <li>Incentivize, reward, coerce, or provide compensation to users for watching a video. A user’s decision to watch a video needs to be their own choice. 
          <ul> 
           <li><strong>Example:</strong> Offering the chance to win a prize or offering financial compensation in exchange for a user watching a video through your API service.</li> 
          </ul></li> 
         <li>Block, modify or replace advertisements that play through the YouTube API service.</li> 
         <li>Allow users to download videos for offline play outside of the YT Premium experience.</li> 
         <li>Offer users the ability to download or separate audio tracks or allow users to modify the audio or video portions of a video. 
          <ul> 
           <li><strong>Example:</strong> Using YouTube’s API to separate or isolate video or audio components from a video. This might include an API service that offers mp3 files of audio that appeared in a video and promotes themselves in this context.</li> 
          </ul></li> 
         <li>Allow for background play of the YouTube video player. 
          <ul> 
           <li><strong>Example:</strong> Using YouTube’s API to allow videos to play even when your API service window is closed or minimized.</li> 
          </ul></li> 
        </ul> 
        <h2 id="your_api_service_must_add_sufficient_independent_value" data-text="Your API service must add sufficient independent value.">Your API service must add sufficient independent value.</h2> 
        <p><strong>What this means:</strong> Don’t use our API to re-create YouTube (e.g. don’t clone, mimic, modify, or reduce standard YouTube features). If your API service mimics any of YouTube’s user experiences, it must add sufficient independent value. Independent value means providing users with added functionality that is not available via the YouTube API today or was not available at the time of API access request, and is otherwise compliant with YouTube’s TOS. More information can be found <a href="/youtube/terms/developer-policies#i.-additional-prohibitions">here</a>.</p> 
        <h3 id="examples_4" data-text="Examples">Examples</h3> 
        <p>If your API service mimics any of YouTube’s user experiences, users need to have a reason to continue to engage with or utilize your API service when you take away what you are getting by accessing YouTube’s API services. You also cannot charge people a fee for services that are offered free of charge on YouTube.</p> 
        <ul> 
         <li><strong>Example of what’s allowed:</strong> A Search engine that lists YouTube videos alongside videos available on other platforms while clearly distinguishing them from what’s on YouTube is a good example of an API service that provides independent value.</li> 
         <li><strong>Example of what’s allowed:</strong> An API service that provides YouTube video captioning services for the hearing impaired is a good example of providing independent value.</li> 
         <li>Don’t use YouTube’s API to create websites or apps or display video search results that make it difficult to distinguish between your website or app and websites or apps created by YouTube. 
          <ul> 
           <li><strong>Example:</strong> Using YouTube’s API to mass aggregate embedded videos, creating an identical copy of YouTube. If a user is likely to mistake your site for YouTube’s, there’s a good chance it violates our TOS. </li> 
          </ul></li> 
        </ul> 
        <h2 id="don’t_allow_users_to_circumvent_youtube_restrictions_or_violate_our_community_guidelines" data-text="Don’t allow users to circumvent YouTube restrictions, or violate our Community Guidelines.">Don’t allow users to circumvent YouTube restrictions, or violate our Community Guidelines.</h2> 
        <p><strong>What this means:</strong> Your service can’t be specifically designed to allow you or your users to get around restrictions YouTube places on their channel. Your API service also can’t let users do activities that violate our <a href="https://www.youtube.com/about/policies/#community-guidelines">community guidelines</a>, <a href="https://www.youtube.com/t/terms">Terms of Service</a>, or YouTube Partner Program. If you use YouTube’s API to allow users to upload videos, you must have them certify that their content complies with the community guidelines. Videos are subject to removal if found to be violative. Your service can also be subject to penalties if it encourages or incites violative behavior. More information can be found <a href="/youtube/terms/api-services-terms-of-service#notices-to-users">here</a>.</p> 
        <h2 id="don’t_spread_api_access_across_multiple_or_unknown_projects" data-text="Don’t spread API access across multiple or unknown projects.">Don’t spread API access across multiple or unknown projects.</h2> 
        <p><strong>What this means:</strong> You can’t create multiple apps/sites or create multiple Google Cloud projects for use across multiple apps/sites to artificially acquire more API quota (aka "sharding") for a single API service or use case. "Use case" is defined as a consistent set of analyses, features, or actions performed via a service. Requests for API quota increase must follow our standard <a href="https://support.google.com/youtube/contact/yt_api_form">process</a>. An application’s developer team is allowed to have separate API keys for test, dev, and prod environments. More information can be found <a href="/youtube/terms/developer-policies#d.-accessing-youtube-api-services">here</a>.</p> 
        <h3 id="examples_5" data-text="Examples">Examples</h3> 
        <ul> 
         <li>Don’t create multiple Google Cloud projects for the same API service or use case in an attempt to deceptively acquire an API quota that is higher than the one your project was assigned.</li> 
         <li>It is acceptable to have a separate API Project for each different use case of your API service. Examples include: 
          <ul> 
           <li>One API project for your iOS app, a separate API Project for your Android app.</li> 
           <li>One API project for a production server, one for a development server.</li> 
           <li>One API project for your user-facing API service, one API project for internal system analytics.</li> 
          </ul></li> 
        </ul>
