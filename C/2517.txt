     <h1 id="getting_started_top">Getting Started<span class="version-indicator">1.5</span></h1> 
     <h2 id="frequently_used_terms">Frequently Used Terms</h2> 
     <br> 
     <p> Below is a list of Frequently Used Terms used throughout this documentation. Any terms specific to MediaFire will be hyperlinked back to this page for further explanation. <br> <br> <b><span id="quick_key">quickkey:</span></b> quickkey is a reference to a file. You can get quickkeys from <a href="/developers/core_api/1.1/folder/#get_content" target="_blank">/folder/get_content.</a> <br> <br> <b><span id="folder_key">folderkey:</span></b> folderkey is a reference to a folder. You can get a list of folderkeys from <a href="/developers/core_api/1.1/folder/#get_content" target="_blank">/folder/get_content</a>. When you use the /folder/get_content API without a folder key defined, it will automatically give you the folderkeys for the folders in your root 'My Files' page. From there, you can retrieve the folderkey of the folder you would like to continue "digging into". Next, update the 'get_content' folder url with the folderkey you would like to inspect. This process can be repeated continually until you reach your desired folderkey. <br> <br> <b><span id="resource">resource:</span></b> resource is a file or folder object in a MediaFire account. <br><br> <b><span id="session_token">session token:</span></b> session token is a token used to authorize user specific transactions. You can get instructions for obtaining a session token at <a href="/developers/core_api/1.1/user/#get_session_token" target="_blank">/user/get_session_token.</a> <br> <br> <b><span id="web_upload">web upload:</span></b> a web upload is when you give MediaFire a URL of an item you would like placed in your account. In return, Mediafire will download the file from the specified URL and place it directly into your My Files (unless you specify a different folderkey). For example: if you give MediaFire the URL www.mediafire.comimages/contact_trie.jpg we will fetch contact_trie.jpg and download it into your account. This is an ideal situation to obtain a file for your account without you having to use your bandwidth to access it. <br> <br> </p> 
     <br>
     <h2 id="best_practices">Best Practices</h2> 
     <br> 
     <p> Below is a list of suggested measures which will ensure your code is using the MediaFire REST API in a more efficient and secure manner: <br> <br> </p>
     <ul> 
      <li> <b>DO</b> use <a href="/developers/core_api/1.1/user/#get_session_token" target="_blank">Session Token 2</a> over type 1. </li> 
      <li> <b>DO</b> use <a href="/developers/core_api/1.1/upload/#resumable" target="_blank">Resumable Uploads</a>. </li> 
      <li> <b>DO</b> use <a href="/developers/core_api/1.1/user/#get_action_token" target="_blank">Action Tokens</a> when fetching large numbers of images using the the Media API and Image Conversion systems. </li> 
      <li> <b>DO</b> use restrictive security features, such as <a href="/developers/getting_started/getting_started/#edit_your_api_security_controls" target="_blank">restrictions by Domains, IP Addresses, and Callback URLs</a>, to improve security for web applications. </li> 
      <li> <b>DO</b> attempt to use the <a href="/developers/core_api/1.1/upload/#instant" target="_blank">Instant Upload</a> feature. </li> 
      <li> <b>DO</b> destroy <a href="/developers/core_api/1.1/user/#get_action_token" target="_blank">Action Tokens</a> when they are no longer necessary. </li> 
      <li> <b>DO</b> use SSL whenever possible. </li> 
      <li> <b>DO</b> use the POST request method instead of GET whenever possible. </li> 
      <li> When performing an upload, <b>DO</b> use <a href="/developers/core_api/1.1/user/#get_action_token" target="_blank">Action Tokens</a> to improve performance. </li> 
      <li> When performing an upload, <b>DO</b> ensure the account you are uploading to has adequate space for the file. </li> 
      <li> <b>DO NOT</b> expose your API keys (when possible). </li> 
     </ul> 
     <p></p>

     <h1 id="api_basics">API Basics</h1> 
     <br>
     <h2 id="call_signature">Call Signature (Session Token 2)</h2> 
     <br> 
     <p> <b>Note:</b> When making an API call using the GET request method, it is permissible to URL-encode the query string. The server will compute the Call Signature with both permutations (encoded and not encoded) looking for a match. <br> <br> <b>Description:</b> The API call signature is required when Session Token version 2 is used. When you call the secure API user/get_session_token and supply 'token_version=2', in addition to 'session_token' you will receive two more values: 'secret_key' and 'time'. You will need to use these values to construct a signature to be passed along with any subsequent API calls that accept a session token. <br> <br> The signature = the Message-Digest (MD5) of the 'secret_key' modulo 256 + 'time' + the URI of the API call. <br> <br> The URI of the call is the URL excluding the signature and the hostname (e.g. "/1.0/file/get_info.php?quick_key=abcdefghijklmno"). <br> <br> After each API call that returns in the response new_key=yes, you will need to generate a new secret_key to use in the construction of the signature for the next API call. The generation of this new key is based on the MINSTD variant of a Linear Congruential Generator (LCG). The formula to use is: New Key = (Old Key * 16,807) modulo 2,147,483,647. <br> <br> <b>Obtaining an App ID and API Key</b> <br> <br> In order to build a signature, you will need to create an app with a unique ID and key. If you do not already have this information, follow these steps: </p>
     <ul> 
      <li> From MediaFire, visit your Account Settings page. </li> 
      <li> Choose "Developers" from the sidebar menu. </li> 
      <li> If you have never created an app before, fill in the requested information and click "Submit". </li> 
      <li> You may need to validate your email address before continuing. </li> 
      <li> Enter a name for your app and click "Create New Application". </li> 
      <li> Copy the generated app ID and API key from the table below. </li> 
     </ul> 
     <br> <b>Notes:</b> Some APIs do not accept a session token and signature (<b>system/get_version</b> for instance). These APIs are designed to be called without needing any type of authentication. If you change your <b>secret_key</b> and construct a signature for these API calls, then your <b>secret_key</b> will become out of sync with the server. You will then be required to re-authenticate with <b>user/get_session_token</b> to acquire a new <b>secret_key</b> and <b>time</b>. Instead of changing your <b>secret_key</b> on each API call, be sure to check each response for a <b>new_key</b> field equal to "yes". This is the appropriate time to change your stored <b>secret_key</b>.

     <p> <br> <b>Making a first call using this Session Token</b> <br> <br> Let's make a call to get the user's personal info. Our API call URI is : /api/1.5/user/get_info.php?session_token=[insert session token here] Therefore, we construct the signature by concatenating the Secret key modulo 256, Time and the URI, then apply MD5 on the result. 9316931 modulo 256 is 67. This will be calculated as follows: signature = MD5('671359061000.8125/api/1.5/user/get_info.php?session_token=[insert session token here) The signature is 938a3a05ba087245fc2c7f96512cdbd8 and the API call would be: <br> http://www.mediafire.com/api/1.5/user/get_info.php?session_token=[insert sample token here]&amp;signature=[insert signature here] <br> </p> 
     <br>

However, the developer will still need to old secret key to generate a new secret key. The New key is calculated as follows: new key = (9316931 * 16807) modulo 2147483647 which is 1970836733 We will make a call to get the root folder's content. Our URI will be: /api/1.5/folder/get_content.php?session_token=[insert session token here]&amp;content_type=files <br> Next, we construct the signature using the new key. 1970836733 modulo 256 is 253.      <h2 id="asynchronous_job_queue">Asynchronous Job Queue</h2> 
     <br> 
     <p> Some API calls require extended processing time, for example copying a folder that has several hundred items in it. For those API calls, the API won't block for the operation to complete, but returns immediately and the operation is performed in the background as an asynchronous job. The API response then returns asynchronous=yes. However, copying a folder with few items in it will be performed synchronously and the API returns asynchronous=no. If an API doesn't return the meta tag asynchronous, the operation is always performed synchronously. Some of the API calls that might be performed asynchronously include, but are not limited to: <br> <br> </p>
     <ul> 
      <li> folder/copy </li> 
      <li> folder/delete </li> 
      <li> folder/move </li> 
      <li> folder/purge, and </li> 
      <li> folder/update </li> 
     </ul>
